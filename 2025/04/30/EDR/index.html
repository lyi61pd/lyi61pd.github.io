简介

EDR是什么

EDR（Endpoint Detection and Response）是一种部署在终端设备上的代理程序，通过监控操作系统生成的事件来识别攻击行为。一旦检测到威胁，EDR会生成警报并将其发送到SIEM或SOAR系统，由人类分析师进行进一步处理。不过EDR也能自行拦截。了解EDR的输入（即它接收到的数据）对于理解其检测机制至关重要

Shellcode加载器

Shellcode加载器的作用是将加密的Shellcode（通常是CobaltStrike、Sliver或Metasploit的Beacon）加载到内存中并执行。加载器的目标是在初始访问阶段（Initial Access）不被EDR检测到。典型的加载过程包括：

  1. 分配一个具有读写权限的内存区域
  2. 将Shellcode复制到该区域并解密
  3. 将内存区域的权限更改为可执行
  4. 执行Shellcode

这种方法的变体包括将Shellcode注入到远程进程中，或通过创建新线程来执行Shellcode。这些操作可能会引起EDR的注意，因此需要谨慎处理。

EDR检测机制

三足鼎立

有三种主要的检测技术：

  1. 文件扫描：文件的签名扫描（如yara）
  2. 内存扫描：进程内存的签名扫描（如yara）
  3. 行为检测：进程的行为（主要是通过操作系统）

文件扫描

大多数C2生成的exe的签名都是能被EDR检测到的，因此第一步是混淆exe中的代码，这样不会触发签名，比如可以参考

https://kleiton0x00.github.io/posts/Harnessing-the-Power-of-Cobalt-Strike-Profiles-for-EDR-Evasion/

或者，使用加载器，可以将payload加密，这种情况下唯一需要从 AV 文件签名扫描中混淆的是 loader 本身

内存扫描

内存扫描可以针对加载器，因为payload必须在内存中解密才能执行。为避免在内存中检测到，进程需要在休眠时加密其内存区域。然后，在 EDR 扫描进程时，内存中不应有任何可疑内容。内存扫描是一项性能密集型作，只有在 EDR 认为值得时才会执行，并不是高频率的。

行为检测

EDR会关联分析进程的行为，比如更改内存权限、创建进程和线程、复制内存等，例如，如果使用加载器绕过 AV，并简单地为 shellcode 分配一个内存区域，那么不会给EDR太多的行为。但是 payload 将被内存扫描器检测到。如果引入内存加密来绕过内存扫描器，那么我们会有更多的可疑行为。也就是说文件扫描、内存扫描、行为检测这三者之间会有一个微妙的平衡，不可兼得。

EDR接收行为数据的来源主要有这些：

  • 用户态（挂钩 API）
  • 内核态（ETW、ETW-TI、内核回调）
  • AMSI内容检测

当添加/删除/更改某些内容时，这些传感器将创建有关系统中发生的事情的事件，例如：

  • 文件
  • 注册表项
  • 进程、线程
  • 内存

需要注意的是，EDR 本身是看不到进程中的数据修改的。或者换句话说，调用 ntdll.dll 的函数 RtlCopyMemory() 的进程可能会生成行为数据，因为 ntdll.dll 可以被hook。但是如果在 for 循环中对字节复制执行相同的作不会产生任何行为数据。

EDR的行为数据主要来自于被hook的ntdll.dll以及内核，主要是通过ETW，用户态的hook可以很容易地被删除绕过，但是这个行为也会生成行为记录，来自内核态的行为数据更可信，无法被删除绕过。

签名扫描（静态特征匹配）

类似传统的杀软，比如通过yara进行扫描

模拟执行

在静态特征匹配之后，EDR会进行模拟执行，模拟执行不是在真实机子上跑的，也不是虚拟化执行，而是EDR有自己实现了一套CPU模拟，会读取.text里的汇编，自己实现了主要的windwos api，然后解析跑程序，类似shellcode加载器，起初shellcode是加密的，此时可能在内存中就解密了，EDR此时进行内存扫描可以匹配特征。不过EDR的模拟执行不是一直运行的，一般会下面这些限制：

  • 运行时间
  • 指令数量
  • API调用数量
  • 内存占用

事件

一旦可执行文件通过了防病毒的签名和仿真扫描，它就可以执行。在执行过程中，进程与操作系统内核交互，例如打开文件、创建网络连接等。这些交互会生成事件，被EDR接收和分析。

EDR通过两种主要渠道接收数据：

  1. 用户模式钩子：拦截用户模式下的API调用
  2. 内核回调：使用ETW（事件跟踪）和内核模式回调函数获取事件

这些事件包括文件、注册表、进程、线程和内存区域的创建、删除和修改等。EDR根据这些事件应用规则，以检测恶意行为

用户态Hook

EDR可以在用户模式下钩住API函数，例如在ntdll.dll中插入钩子。这些钩子可以被攻击者绕过，例如使用直接系统调用或恢复原始的ntdll.dll。然而，绕过钩子的行为本身可能会引起EDR的注意。

ntdll.dll是windows最底层的库，这个库可以帮我们进行syscall调用，其他的WindowsAPI或库，最后都会直接或间接地调用ntdll.dll，所以EDR都会在ntdll.dll进行hook，而其他的链接库则不一定会做hook，毕竟最终还会过来。

这是通过使用内核回调 （PsSetCreateProcessNotifyRoutine）在早期阶段创建新进程时收到通知来实现的，然后将 DLL 注入到进程中（如 amsi.dll），创建一个 APC，该 APC 调用 DLL 的函数，该函数hook原始 ntdll.dll 函数（或通常为 DLL 导入地址表）以通过使用异步过程调用（kKAPC 注入）绕道进入 amsi.dll。

因此，在hook ntdll.dll 后，每个函数调用都会被 amsi.dll 拦截。

用户态的hook并不可靠，可以通过以下方式绕过：

  • 直接系统调用（避免调用 ntdll.dll）
  • 间接系统调用（调用 ntdll.dll 函数，但在hook之后）
  • unhook

内核分析

Windows操作系统提供了多种内核级别的通知回调机制，例如PsSetCreateProcessNotifyRoutine、PsSetCreateThreadNotifyRoutine等。这些回调函数可以通知EDR有关进程和线程的创建、终止等事件。由于这些回调是在内核中实现的，攻击者难以绕过。

EDR还可以使用ETW（事件跟踪）来获取更详细的事件信息，例如进程加载的模块、内存映射等。这些信息有助于EDR进行更深入的行为分析。

内存分析

进程的内存区域可以分为两类：

  1. 映像区域（IMAGE/Backed）：这些区域是从磁盘上的可执行文件映射到内存中的，通常被认为是可信的
  2. 私有区域（USER/PRIVATE/Unbacked）：这些区域是进程在运行时分配的，可能包含Shellcode或其他恶意代码，因此被认为是不可信的

EDR可以通过分析内存区域的属性，例如权限、类型等，来识别潜在的恶意行为。例如，具有读、写和执行权限的私有内存区域（RWX）可能是Shellcode的存储位置。

由于内存扫描是性能密集型操作，通常只有在EDR检测到可疑行为时才会触发。攻击者可以通过在进程空闲时加密内存中的Shellcode，以规避内存扫描。

查询进程信息

EDR可以查询进程的详细信息，例如通过GetProcessInformation或NtQueryInformationProcess获取进程的PEB（Process Environment Block）和EPROCESS结构。这些结构包含进程的映像名称、命令行参数、父进程ID等信息。攻击者可以尝试伪造这些信息，但EDR可以通过比较事件中的信息和实际的内核数据结构来检测伪造行为。

调用栈分析

EDR可以分析进程的调用栈，以识别异常行为。例如，一个正常的调用栈可能包括应用程序函数、kernel32.dll中的API函数、ntdll.dll中的系统调用等。如果调用栈中出现了来自非映像区域的函数。

Threads can be sleeping for different reasons. Investigating the state, and how the thread got there due his callstack, we find indicators for sleeping beacons, or memory encryption.

If memory encryption is being used, the thread is usually put to sleep by calling either:

  • Source in non-backed memory regions

性能影响

EDR 的性能至关重要，假如通过npm安装10000个包，EDR要是给他们全部扫描一遍导致速度缓慢，那用户可能就放弃windows使用apple了，因此对EDR来说，不是针对所有程序都要进行全方位扫描的。

性能影响大概是这样排级

对性能的影响（越小代表越少） 操作
1 单个行为事件
3 行为事件关联
10 查询进程信息
100 内存扫描
1000 文件扫描

什么时候会触发内存扫描呢？

API 触发扫描 备注
VirtualAlloc() 不触发 除了RWX以外，都是正常操作，不会触发
WriteProcessMemory() 不触发 正常操作
memcpy() 不触发 不清楚啥情况，博客中说的是对EDR不可见
VirtualProtect 看情况 RWX或者是从RW到RX的情况可能会触发，其他情况不会触发
CreateRemoteThread() 触发 很可能会触发
